Como Funciona

O CRM usa um motor de regras configuraveis para mover leads automaticamente entre os estagios do pipeline. Em vez de logica hardcoded, voce define regras na pagina de Configuracoes do CRM.

Conceito chave: Quando qualquer dado relevante de um lead muda (historico de contato, status, servico Hubsoft, tag), o motor avalia TODAS as regras e move o lead para o estagio correto automaticamente.

O que dispara a avaliacao?

EventoQuando
HistoricoContato criadoAPI registra novo historico para o lead
LeadProspecto atualizadostatus_api, documentacao_validada, etc. mudam
ServicoClienteHubsoft salvoSincronizacao com Hubsoft detecta mudanca
Tags alteradasTags adicionadas ou removidas da oportunidade

Tipos de Condicao

Cada regra e composta por uma ou mais condicoes. Estes sao os tipos disponiveis:

TipoDescricaoExemplo de Uso
historico_status Lead possui historico de contato com determinado status Status = instalacao_agendada
lead_status_api LeadProspecto.status_api tem valor especifico Status API = processado
lead_campo Campo booleano do lead (documentacao_validada, contrato_aceito, etc.) Documentacao Validada = Sim
servico_status Lead tem servico no Hubsoft com status_prefixo especifico Status = servico_habilitado
tag Oportunidade do lead possui determinada tag Tag = Comercial
converteu_venda Lead tem historico com converteu_venda=True Converteu Venda = Sim

Operadores

OperadorDescricao
igualValor deve ser exatamente igual
diferenteValor deve ser diferente
existeCampo/conjunto possui algum valor (verdadeiro)
nao_existeCampo/conjunto esta vazio (falso)

Logica de Avaliacao

Dentro de uma mesma regra: E (AND)

Todas as condicoes devem ser verdadeiras para a regra ser satisfeita.

// Regra: "Aguardando Instalacao com Doc" Condicao 1: servico_status igual "aguardando_instalacao" E Condicao 2: lead_campo documentacao_validada igual true // Ambas devem ser verdadeiras para mover o lead

Entre regras do mesmo estagio: OU (OR)

Basta UMA regra ser satisfeita para o lead ser movido para o estagio.

// Estagio: "Cliente Ativo" Regra 1: servico_status igual "servico_habilitado" OU Regra 2: converteu_venda igual true // Se qualquer uma bater, o lead vai para Cliente Ativo
Prioridade: Estagios sao avaliados na ordem do pipeline (campo ordem). O primeiro estagio cuja regra bater vence. Estagios finais (Ganho/Perdido) nao sao reavaliados.

API: Registrar Historico de Contato

Registra um novo historico de contato para um lead. O motor de regras e disparado automaticamente apos o registro.

POST /api/historicos/registrar/
Cria um novo HistoricoContato vinculado ao lead.

Payload

{ "telefone": "5511999990000", // obrigatorio "status": "instalacao_agendada", // obrigatorio - qualquer valor da tabela acima "lead_id": 123, // opcional - vincula ao lead diretamente "observacoes": "Instalacao agendada para 05/04", "origem_contato": "whatsapp", "nome_contato": "Joao Silva" }

Resposta (sucesso)

{ "sucesso": true, "id": 456, "dados": { ... } }
Efeito automatico: Apos o registro, o signal avaliar_regras_apos_historico dispara o motor de regras. Se alguma regra configurada usar historico_status = instalacao_agendada, o lead sera movido automaticamente.

API: Atualizar Status do Lead

Atualiza o status_api de um lead. O motor de regras e disparado automaticamente.

POST /api/leads/status/
Atualiza o status_api do LeadProspecto pelo lead_id.

Payload

{ "lead_id": 123, // obrigatorio "status_api": "em_instalacao" // obrigatorio - qualquer valor da tabela acima }

Resposta

{ "sucesso": true, "lead_id": 123, "status_api": "em_instalacao" }

API: Gerenciar Tags

Adiciona e/ou remove tags de um lead. Tags que nao existem sao criadas automaticamente. O motor de regras e disparado.

POST /api/leads/tags/
Adiciona e/ou remove tags da OportunidadeVenda pelo lead_id.

Payload

{ "lead_id": 123, // obrigatorio "tags_add": ["Comercial", "Urgente"], // tags a adicionar "tags_remove": ["Endereco"] // tags a remover }

Resposta

{ "sucesso": true, "lead_id": 123, "oportunidade_id": 78, "tags_adicionadas": ["Comercial", "Urgente"], "tags_removidas": ["Endereco"], "tags_atuais": ["Comercial", "Documental", "Urgente"] }

API: Atualizar Historico Existente

POST /api/historicos/atualizar/
Atualiza um HistoricoContato existente por criterio de busca.

Payload

{ "termo_busca": "protocolo_atendimento", // campo para buscar "busca": "ATD-2026-001", // valor para buscar "status": "proposta_aceita", // novos dados "observacoes": "Cliente aceitou proposta de 100Mb" }

Exemplo: Instalacao Agendada

Cenario: Lead agenda instalacao durante atendimento

Regra configurada no estagio "Aguardando Instalacao":

Condicao 1: historico_status igual "instalacao_agendada" E Condicao 2: lead_campo documentacao_validada igual true

O que o fluxo N8N faz:

1
Registra o historico com status instalacao_agendada
POST /api/historicos/registrar/ { "telefone": "5511999990000", "lead_id": 123, "status": "instalacao_agendada", "observacoes": "Instalacao agendada para 10/04 as 14h" }
2
Motor de regras avalia automaticamente:
- Lead tem historico com status instalacao_agendada? SIM
- Lead tem documentacao_validada = true? SIM
- Resultado: Lead movido para "Aguardando Instalacao"

Exemplo: Proposta Enviada

Cenario: Lead recebe proposta comercial

Regra configurada no estagio "Proposta Enviada":

Condicao 1: historico_status igual "proposta_enviada"

Fluxo N8N:

// Passo 1: Registrar historico POST /api/historicos/registrar/ { "telefone": "5511999990000", "lead_id": 456, "status": "proposta_enviada" } // Passo 2 (opcional): Atribuir tag POST /api/leads/tags/ { "lead_id": 456, "tags_add": ["Comercial"] }

O lead sera movido automaticamente para o estagio "Proposta Enviada".

Exemplo: Cliente Ativo

Cenario: Servico e habilitado no Hubsoft

Regra configurada no estagio "Cliente Ativo":

Regra 1: servico_status igual "servico_habilitado" OU Regra 2: converteu_venda igual true

Neste caso, nenhuma chamada de API e necessaria. Quando a sincronizacao do Hubsoft detectar que o status_prefixo mudou para servico_habilitado, o motor de regras move o lead automaticamente para "Cliente Ativo".

Exemplo: Fluxo Completo N8N

Jornada completa de um lead pelo pipeline via API

1
Lead entra (Novo Lead) - Criado automaticamente pelo sistema.
2
Qualificacao - Documentacao enviada
POST /api/historicos/registrar/ { "telefone": "5511999990000", "lead_id": 100, "status": "documentacao_enviada" }
Se houver regra com historico_status = documentacao_enviada, move para o estagio configurado.
3
Negociacao - Proposta aceita + contrato assinado
// Registrar aceitacao POST /api/historicos/registrar/ { "telefone": "5511999990000", "lead_id": 100, "status": "proposta_aceita" } // Atualizar status do lead POST /api/leads/status/ { "lead_id": 100, "status_api": "processado" } // Registrar contrato POST /api/historicos/registrar/ { "telefone": "5511999990000", "lead_id": 100, "status": "contrato_assinado" }
4
Fechamento - Instalacao agendada
POST /api/historicos/registrar/ { "telefone": "5511999990000", "lead_id": 100, "status": "instalacao_agendada" }
5
Cliente Ativo - Automatico quando Hubsoft sincroniza o servico como servico_habilitado.

Comandos Django

python manage.py revalidar_pipeline
Reavalia todas as oportunidades ativas contra as regras configuradas e move quando necessario.
# Simular sem gravar python manage.py revalidar_pipeline --dry-run # Filtrar por tipo de estagio python manage.py revalidar_pipeline --estagio negociacao # Limitar quantidade python manage.py revalidar_pipeline --limit 50
python manage.py popular_crm
Cria oportunidades para leads existentes e depois reavalia regras automaticamente.
python manage.py popular_crm --dry-run python manage.py popular_crm --limpar # apaga oportunidades antes
